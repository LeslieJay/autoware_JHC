## 1. 概述

Autoware 中处理障碍物的模块分布在 **Behavior Planning（行为层）** 和 **Motion Planning（运动层）** 两个层级：

| 层级 | 模块 | 职责 |
|------|------|------|
| **行为层** | Static Obstacle Avoidance | 车道内横向避让静态障碍物 |
| **行为层** | Avoidance by Lane Change | 通过变道避让静态障碍物 |
| **行为层** | Dynamic Obstacle Avoidance | 动态调整路径避让移动障碍物 |
| **运动层** | Obstacle Cruise Planner | 纵向速度规划（停车/巡航/减速） |

**核心设计理念**：行为层负责 **"怎么绕"**，运动层负责 **"绕不了就停/减速"**。

## 3. Static Obstacle Avoidance 模块

### 3.1 模块功能

**核心功能**：在当前车道内通过 **横向偏移（shift）** 避让静止或缓慢移动的障碍物。

**适用场景**：
- 路边停放的车辆
- 施工区域的锥桶
- 静止的行人/自行车
- 掉落物

### 3.2 工作流程

```
输入: 障碍物列表 + 当前路径
         │
         ▼
┌─────────────────────────┐
│  1. 目标过滤            │  判断哪些障碍物需要避让
│     (Target Filtering)   │
└───────────┬─────────────┘
            ▼
┌─────────────────────────┐
│  2. 空间检查            │  计算避让所需空间
│     getAvoidMargin()     │
└───────────┬─────────────┘
            │
    ┌───────┴───────┐
    ▼               ▼
 空间足够         空间不足
    │               │
    ▼               ▼
┌─────────────┐ ┌─────────────────────────┐
│ 3. 生成避让 │ │ 标记为                   │
│    移位线   │ │ INSUFFICIENT_DRIVABLE   │
│ (Shift Line)│ │ _SPACE                   │
└──────┬──────┘ └───────────┬─────────────┘
       │                    │
       ▼                    ▼
┌─────────────┐      交给 Obstacle Cruise
│ 4. 输出避让 │      Planner 处理（停车）
│    路径     │
└─────────────┘
```

```cpp

  const ObjectData & object,
  const std::shared_ptr<AvoidanceParameters> & parameters)
{
  // 计算硬性横向距离限制 = 道路边界到自车的最大可用空间
  const auto hard_lateral_distance_limit = ...;
  
  // 计算最小避让裕度 = 安全距离 + 车宽/2
  const auto min_avoid_margin = ...;
  
  // 核心判断：如果可用空间 < 最小避让裕度，返回 nullopt
  if (hard_lateral_distance_limit < min_avoid_margin) {
    return std::nullopt;  // 空间不足，无法避让
  }
  
  // 空间足够，返回实际避让裕度
  return std::min(soft_lateral_distance_limit, max_avoid_margin);
}
```

## 4. Avoidance by Lane Change 模块

### 4.1 模块功能

**核心功能**：当存在可变道的相邻车道时，通过 **变道** 来避让当前车道上的静态障碍物。

**设计思想**：复用 Static Obstacle Avoidance 的 **目标过滤逻辑** + Lane Change 模块的 **路径生成逻辑**，但有独立的触发条件。

### 4.2 触发条件

执行请求函数 `isExecutionRequested()` 检查三个条件：

```cpp
bool AvoidanceByLCInterface::isExecutionRequested() const
{
  return module_type_->isLaneChangeRequired() &&   // 条件1
         module_type_->specialRequiredCheck() &&    // 条件2
         module_type_->isValidPath();               // 条件3
}
```

| 条件 | 函数 | 说明 |
|------|------|------|
| **条件1** | `isLaneChangeRequired()` | 存在可变道的相邻车道 |
| **条件2** | `specialRequiredCheck()` | 当前车道障碍物数量 >= `execute_object_num` |
| **条件3** | `isValidPath()` | 生成的变道路径有效 |

### 4.3 核心判断逻辑

**文件位置**:
```
src/universe/autoware_universe/planning/behavior_path_planner/
  autoware_behavior_path_avoidance_by_lane_change_module/src/scene.cpp
```

**`specialRequiredCheck()` 函数** (第152-200行):

```cpp
bool AvoidanceByLaneChange::specialRequiredCheck() const
{
  // 统计当前车道上需要避让的目标数量
  size_t num_of_avoidance_targets = 0;
  for (const auto & object : avoidance_data_.target_objects) {
    if (object.is_avoidance_target) {
      num_of_avoidance_targets++;
    }
  }
  
  // 核心判断：目标数量 >= execute_object_num (默认1)
  return num_of_avoidance_targets >= parameters_->execute_object_num;
}
```

## 5. 两个模块的协作机制

### 5.1 Planner Manager 的模块选择

两个模块并非串联关系，而是由 **Planner Manager** 统一管理，通过 **优先级** 和 **isExecutionRequested()** 决定执行哪一个。

**核心选择逻辑** (planner_manager.cpp):

```cpp
std::vector<SceneModulePtr> SubPlannerManager::getRequestModules(...) 
{
  for (const auto & manager_ptr : manager_ptrs_) {
    // 遍历所有模块管理器
    if (manager_ptr->isExecutionRequested(previous_module_output)) {
      request_modules.emplace_back(manager_ptr->getIdleModule());
    }
  }
  // 按优先级排序
  sortByPriority(request_modules);
  return request_modules;
}
```

### 5.3 场景对照表

| 场景 | Static Obstacle Avoidance | Avoidance by Lane Change |
|------|--------------------------|-------------------------|
| 单个障碍物 + 有相邻车道 | 可能被跳过 | **优先执行** |
| 单个障碍物 + 无相邻车道 | **执行** | 不执行 |
| 车道内空间不足 | **不生成路径** (交给OCP) | 不受影响 |
| 多个连续障碍物 | 逐个避让 | **优先执行** (一次变道) |
| 障碍物距离太近无法变道 | **执行** | 不执行 |

## 6. Obstacle Cruise Planner 的兜底作用

### 6.1 功能定位

**Obstacle Cruise Planner (OCP)** 是运动层的障碍物处理模块，负责 **纵向速度规划**，作为行为层避让的 **兜底方案**。

### 6.2 三种处理模式

| 模式 | 触发条件 | 行为 |
|------|----------|------|
| **Stop** | 障碍物在路径上且无法绕行 | 完全停车 |
| **Cruise** | 前方有移动障碍物 | 保持安全距离跟随 |
| **Slow Down** | 路径附近有障碍物 | 减速通过 |

### 6.3 与行为层的协作

```
行为层输出: Path (可能包含避让偏移，也可能没有)
              │
              ▼
┌─────────────────────────────────────────────────┐
│         Obstacle Cruise Planner                  │
│                                                  │
│  ┌─────────────────────────────────────────┐    │
│  │ 检查路径上是否仍有障碍物                  │    │
│  └─────────────────┬───────────────────────┘    │
│                    │                            │
│            ┌───────┴───────┐                    │
│            ▼               ▼                    │
│       有障碍物         无障碍物                  │
│            │               │                    │
│            ▼               ▼                    │
│  ┌─────────────────┐ ┌─────────────┐           │
│  │ 插入停车/减速点  │ │ 保持原速度   │           │
│  └─────────────────┘ └─────────────┘           │
└─────────────────────────────────────────────────┘
              │
              ▼
输出: Trajectory (含完整速度规划)
```

### 6.4 处理 "空间不足" 场景

当 Static Obstacle Avoidance 标记障碍物为 `INSUFFICIENT_DRIVABLE_SPACE` 时：

1. **行为层不生成避让路径** - 路径仍经过障碍物位置
2. **OCP 检测到路径上有障碍物** - 触发 Stop 模式
3. **车辆停在障碍物前** - 等待人工干预或障碍物移除

## 附录: 常见问题

### Q1: 为什么车辆有时不避让障碍物而是直接停车？

**原因**: 避让空间不足。`getAvoidMargin()` 判断 `hard_lateral_distance_limit < min_avoid_margin` 时返回 `nullopt`，障碍物被标记为 `INSUFFICIENT_DRIVABLE_SPACE`，交给 OCP 处理停车。

### Q2: 如何调整避让激进程度？

调整 `static_obstacle_avoidance.param.yaml` 中的：
- `hard_margin` / `soft_margin` - 安全边距
- `max_right_shift_length` / `max_left_shift_length` - 最大偏移距离

### Q3: 如何让车辆优先变道而非车道内避让？

调整 `avoidance_by_lane_change.param.yaml` 中的：
- `execute_object_num: 1` - 设为1表示只要有1个障碍物就变道
- `execute_only_when_lane_change_finish_before_object: false` - 放宽变道完成位置约束

### Q4: 两个模块的优先级如何配置？

在 Behavior Path Planner 的配置文件中，通过模块注册顺序决定优先级。先注册的模块优先级更高。
